## Methodology

Each candidate was checked in its defining source and with repository-wide searches across production code and tests. The review also checked base-class/inheritance behavior, Qt virtual event dispatch, signal/slot and callback wiring, layout/parent ownership, cross-module coordinator protocols, test seams, dynamic `getattr`/`setattr` patterns, package re-exports, and public/private API intent. Audit-provided reference counts were not treated as proof. `FALSE_POSITIVE` means the symbol is used or is a framework-required override; `CONFIRMED` means the reported binding/API is genuinely removable even when the expression that creates a wired widget must remain; `UNCERTAIN` is reserved for evidence that cannot safely distinguish those cases.

## Cross-cutting patterns

- **10 Qt-dispatched overrides are false positives.** Vulture cannot see virtual dispatch for `nextCheckState`, painting, or drag/drop events. These should be allowlisted by qualified method/base class rather than removed.
- **21 test-facing queries and UI handles are false positives.** They form narrow observability/control seams for models, panels, dialogs, settings pages, and badges. Retain them and document that intent if the audit must remain clean.
- **18 settings-widget attributes are confirmed dead bindings.** The widgets themselves are live because their builders connect signals and add them to Qt layouts; cleanup should drop only the unnecessary `self._... =` names, not the builder calls.
- **Coordinator extraction left stale forwarding APIs.** Six `MediaWorkspace` wrappers plus `SettingsTab._set_key_status` are bypassed by the active coordinator paths; `MediaWorkspace.splitter` is likewise unconsumed.
- **Repeated field writes can multiply one defect.** The three `_poster_pixmap` records are one write-only field lifecycle, while the two `QPointF` records are a true Ruff/Vulture duplicate and require only one source edit.
- No assigned record depended on unresolved reflection, generated UI names, package re-export, dataclass/model implicit field consumption, or undocumented external API intent; therefore no `UNCERTAIN` verdict was needed.
